The problem statement

Using AI in VS Code / GitHub is intuitive — integrating AI with automation wasn’t

What’s easy (developer UX)

  • Ask Copilot in VS Code and get help immediately
  • Tag AI in GitHub PRs for review or suggestions
  • Feedback loop is tight: you control files + commands

Key property: the IDE/PR UI already provides guardrails (scoped files, human in the loop).

What’s hard (automation)

  • Every integration needs safe execution + permission boundaries
  • Tooling differs per use case (bug fix vs log analysis vs triage)
  • Without careful design, an LLM can explode output or run risky commands
  • Long tasks hit context limits and drift without memory/compaction

Observed reality: integrating AI with automation is more “systems engineering” than prompt-writing.

Pattern we kept repeating: only the prompt + a few special tools change — but we kept rebuilding the rest (loop, sandbox, safety, context) for each AI integration.

So what is Microbots?

A reusable foundation for building secure automation agents

Microbots turns an LLM into an automation agent by providing: a strict action schema, a sandboxed environment, pluggable tools, and context management.

In this repo: the core loop lives in src/microbots/MicroBot.py.

The agent loop (real code)

In src/microbots/MicroBot.py, run() executes:

  • LLM returns { task_done, thoughts, command }
  • MicroBot validates command safety
  • Environment executes command, returns output
  • Output is fed back to LLM as the next message

Hard constraints (shared)

  • No interactive editors (vim/nano)
  • Avoid massive outputs (no ls -R, unbounded find)
  • Iteration/time limits (max iterations + timeout)
  • Optional context compaction via summarize_context

def _is_safe_command(self, command: str) -> tuple[bool, Optional[str]]:
		explanation = self._get_dangerous_command_explanation(command)
		is_safe = explanation is None
		return is_safe, explanation

Why a strict action schema?

Automation needs predictability: parsing free-form text is fragile and unsafe

Benefits

  • Every step yields exactly one command
  • Clear stop condition (task_done=true)
  • Enables command validation and audit logs
  • Makes tool execution deterministic

Common failure modes without schema

  • Multiple commands at once (hard to attribute failures)
  • Chatty explanations mixed with commands
  • “Assumed success” without verification
  • Large unbounded output causes context blow-ups

Microbots’ schema

{
	"task_done": false,
	"thoughts": "Need to inspect MicroBot.py",
	"command": "sed -n '1,200p' src/microbots/MicroBot.py"
}

Microbots nuance

Notice the example uses bounded reading (sed -n '1,200p') rather than cat / recursive browsing. That guidance is enforced in the shared system prompt.

Show: shared system prompt guidance (bounded exploration) 
# src/microbots/MicroBot.py (system_prompt_common excerpt)
NEVER use commands that produce large amounts of output...
Use specific patterns: 'find <path> -name "*.c" -maxdepth 2'
Avoid: 'ls -R', 'rm -rf', 'tree', or 'find' without depth limits
You cannot run any interactive commands like vim, nano, etc.

Component 2: Prompts

System prompt defines safe workflow; task prompt defines the specific problem in hand.

Prompts: system vs task

Microbots uses a reusable “common” system prompt + bot-specific add-ons

System prompt (general instructions)

  • Define response format + command rules
  • Specify safe exploration patterns
  • Define tools like summarize_context
  • State “no human in the loop” constraints

Microbots centralizes this in system_prompt_common.

Task prompt (specific work)

  • Describe the immediate objective and success criteria
  • Point at relevant files/logs with mounted paths
  • Limit scope (don’t explore unrelated code)
  • Ask for verification (tests, reproduction steps)

Microbots encourages small steps: one command, observe output, iterate.

Show: where system_prompt_common lives (real code)

...

{llm_output_format_str}
...

		1. summarize_context: ...
"""

Prompt design checklist (what to include)
  • Goal: the outcome in one sentence
  • Scope: which directories are allowed / mounted paths
  • Constraints: output limits, non-interactive, safe commands
  • Verification: how to prove success (tests, reproduction, diff)
  • Stop rule: when to set task_done=true

Example: turning a vague ask into a good task prompt

Interactive: expand and compare prompts

Vague task prompt (bad for automation) 
Fix the failing tests.

Problems: no scope, no verification strategy, encourages random exploration.

Microbots-style task prompt (bounded + verifiable) 
Goal: Fix the failing unit tests in this repo.

Scope:
- You may inspect files only under /workdir/<mounted_repo>.

Workflow:
1) Reproduce the failure by running pytest with a targeted subset.
2) Identify the smallest root cause.
3) Apply a minimal fix.
4) Re-run the same tests to verify.

Constraints:
- Avoid unbounded output and recursive listing.
- One command at a time.

Success:
- The previously failing tests pass.

Microbots insight: Good prompts are not longer — they are more structured. Structure reduces random wandering, output explosions, and unsafe actions.

Component 3: Tools

From “cat entire file” to carefully bounded repo navigation — plus special tools like cscope, ctags, gdb.

Tooling evolved: bounded exploration

Early LLMs tried to read everything — Microbots forces safe, incremental navigation

Then: “read the repo”

# Anti-patterns
cat src/microbots/MicroBot.py
ls -R
find . -name "*.py"

These cause huge output, timeouts, and context overflow.

Now: “slice & navigate”

# Safer patterns
sed -n '1,200p' src/microbots/MicroBot.py
grep -R "def _is_safe_command" -n src/microbots
find src/microbots -maxdepth 3 -name "*.py"

Microbots includes both prompt guidance and runtime blocks for dangerous patterns.

Show: Microbots runtime blocking logic (pattern-based) 
# src/microbots/MicroBot.py
dangerous_checks = [
	{ pattern: r"\\bls ... -R", alternative: "ls <specific-dir>" },
	{ pattern: r"\\btree\\b", alternative: "find ... -maxdepth 2" },
	{ pattern: r"\\brm ... -rf", alternative: "rm <specific-file>" },
	{ pattern: r"\\bfind\\b(?!.*-maxdepth)", alternative: "find ... -maxdepth 2" },
]

Microbots tool model: Tool + YAML

Tools are plugins: install, verify, setup, and teach the LLM how to use them

Tool dataclass

src/microbots/tools/tool.py

@dataclass
class Tool:
	name: str
	description: str
	usage_instructions_to_llm: str
	install_commands: List[str]
	verify_commands: Optional[List[str]]
	setup_commands: Optional[List[str]]
	env_variables: Optional[List[str]]
	files_to_copy: Optional[List[EnvFileCopies]]

Most integrations only need to add a YAML file and pass it as additional_tools.

Lifecycle

  • Install: run commands in sandbox
  • Verify: fail fast if missing
  • Setup: build indexes (e.g., cscope DB)
  • Teach: append usage instructions into system prompt
Show: how tool usage instructions reach the LLM 
# src/microbots/MicroBot.py (_create_llm)
system_prompt_with_tools = self.system_prompt
for tool in self.additional_tools:
	if tool.usage_instructions_to_llm:
		system_prompt_with_tools += "\n\n" + tool.usage_instructions_to_llm

Tooling challenge solved: Microbots makes tool installation and “how to use it” part of the same artifact, so automation doesn’t rely on tribal knowledge.

Component 4: Environment

The environment is where LLM actions happen — Microbots uses Docker sandboxing, explicit mounts, and command execution via a shell server.

Why a sandbox is non-negotiable

Automation agents must be least-privilege and network-restricted by default

Risks without sandboxing

  • Accidental destructive commands on host filesystem
  • Unbounded network access (data exfiltration risk)
  • Secrets exposure in environment variables
  • Hard to audit: “what did the agent do?”

Microbots environment guarantees

  • Every run creates a Docker container
  • Mounts are explicit: READ_ONLY vs READ_WRITE
  • Commands validated before execution
  • Logs available for audits
  • Environments can be torn down after completion
Show: LocalDockerEnvironment creates container + volumes 
# src/microbots/environment/local_docker/LocalDockerEnvironment.py

Mounts: explicit permissions and paths

Microbots represents mounts with a first-class object: Mount

Mount object

src/microbots/extras/mount.py

@dataclass
class Mount:
	host_path: str
	sandbox_path: str
	permission: PermissionLabels
	mount_type: MountType = MountType.MOUNT

# MountType: MOUNT | COPY

This prevents “hidden” access: you can’t accidentally hand the agent more than you intended.

Example: LogAnalysisBot uses COPY mount

src/microbots/bot/LogAnalysisBot.py

file_mount_info = Mount(
	file_name,
	LOG_FILE_DIR,
	PermissionLabels.READ_ONLY,
	MountType.COPY,
)

return super().run(
	task=f"Analyze the log file `{file_mount_info.sandbox_path}`",
	additional_mounts=[file_mount_info],
)

This is a clean pattern: code is mounted (RO), logs are copied in, analysis happens safely.

Execution: LLM plans, sandbox executes

Microbots runs commands via an HTTP shell server inside the container

Environment.execute()

src/microbots/environment/local_docker/LocalDockerEnvironment.py

response = requests.post(
	f"http://localhost:{self.port}/",
	json={"message": command},
	timeout=timeout,

return CmdReturn(
	stdout=output.get("stdout", ""),
	stderr=output.get("stderr", ""),

Why this design?

  • LLM never gets a host shell
  • Commands become explicit audit events
  • Timeouts and recovery can be centralized
  • One command at a time reduces state confusion

Component 5: Context

Long-running automation hits token limits. Success requires compaction, memory, and decomposition into sub-tasks.

Context is the make-or-break component

Even great prompts fail if the agent can’t manage state over many steps

Common failure modes

  • Token overflow from repeated command outputs
  • Drift: agent forgets original goal or constraints
  • Looping on the same failing command
  • Inability to switch from exploration → execution

Microbots approach

  • Strict one-command-per-step loop
  • Iteration + timeout caps to avoid runaway
  • Built-in compaction tool: summarize_context
  • Encourage decomposition using multiple bots in a pipeline

Rule of thumb: treat context like a constrained resource — you spend it on high-signal outputs only.

Microbots

Compaction: summarize_context

Microbots lets the LLM compress its history without changing the system prompt rules

How the LLM calls it

{
	"task_done": false,
	"thoughts": "Context is too long; summarize and continue",
	"command": "summarize_context 2 \"Found root cause in MicroBot.run; next patch tool validation\""
}

The command is special-cased in the agent loop (it is not executed in the shell).

How MicroBot handles it

# src/microbots/MicroBot.py
if llm_response.command.startswith("summarize_context"):
	parsed = self._parse_summarize_context_command(...)
	last_msg = self.llm.summarize_context(...)
	llm_response = self.llm.ask(last_msg["content"])

Decomposition: break work into sub-tasks

You can chain bots (read → write → verify) instead of one giant agent run

Pipeline sketch

from microbots.bot.ReadingBot import ReadingBot
from microbots.bot.WritingBot import WritingBot

reader = ReadingBot(
	model="azure-openai/gpt-5",
	folder_to_mount="/repo",
)
diagnosis = reader.run("Find why tests fail")

writer = WritingBot(
	model="azure-openai/gpt-5",
	folder_to_mount="/repo",
)
fix = writer.run(f"Fix: {diagnosis.result}")

Why this improves context

  • Each bot has a focused prompt and smaller state
  • Different permissions per phase (RO → RW)
  • Tool sets can differ per phase
  • Results become concise hand-offs between steps

This is “sub-agents” in practice: multiple small agents with tight contracts.
