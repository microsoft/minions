What is MicroBots?

Secure AI agents that bridge LLMs and your automation workflows

The Problem

  • LLMs are powerful but lack safe execution environments
  • Giving AI direct filesystem access is a security risk
  • No standardized way to plug AI into CI/CD securely
  • Manual code fixes don't scale in large codebases

The MicroBots Solution

  • Containerized Docker sandbox for every task
  • Explicit read-only or read-write permission per mount
  • LLM-agnostic: OpenAI, Anthropic, or local Ollama
  • Pluggable tools installable via simple YAML configs
  • Purpose-built bot types for specific workflows

One-liner:  MicroBots securely mounts your code into an isolated sandbox, lets an LLM inspect or modify it via shell commands, and returns structured results — all with least-privilege access controls and defense-in-depth security.

Architecture Overview

Three-layer design: LLM → Bot Agent → Sandbox Environment

LLM Layer
Azure OpenAI • Anthropic Claude • Ollama Local
↓  JSON: { task_done, thoughts, command }  ↓
MicroBot Agent Layer
ReadingBot • WritingBot • BrowsingBot • LogAnalysisBot • CustomBot
↓  HTTP POST (FastAPI)  ↓
Docker Sandbox Environment
Overlay FS (RO) • Direct Mount (RW) • Shell Server • Tool Runtime

Agentic Loop

The bot sends a task to the LLM. The LLM responds with a shell command. The bot executes it in the sandbox and feeds the output back. This loop continues until the LLM sets task_done=true or limits are hit.

Safety Controls

Dangerous commands (rm -rf, ls -R, find without depth limit) are automatically blocked before execution with explanatory feedback to the LLM.

Context Management

Built-in summarize_context capability lets the LLM compress its conversation history to stay within token limits during long tasks.

Secure Sandboxing — Security First

Every AI action is isolated, permission-gated, and command-validated

🐳 Container Isolation

Every bot task spawns a fresh Docker container. The host filesystem is never directly exposed. Containers are destroyed after task completion — no persistent attack surface.

# Isolated container per task
image: shell_server:latest
# Communication via HTTP only
MicroBot → HTTP POST → Container
# Auto-cleanup on completion

🔒 Permission Boundaries

Mount permissions are explicit and enforced. ReadingBot uses overlay FS — writes go to a temp layer and are discarded. WritingBot scopes writes to the mounted directory only.

# ReadingBot: overlay FS
mount: /repo → /workdir  [ro]
overlay: upper=tmpfs

# WritingBot: direct mount
mount: /repo → /workdir  [rw]
# scoped to mounted path only

🛡 Command Validation

Every LLM-generated command is validated before execution. Destructive patterns are blocked with explanatory feedback. The LLM is guided to use safe alternatives.

# Blocked commands:
rm -rf /       # recursive delete
ls -R          # recursive listing
find / -name * # unbounded find

# Safe alternative returned:
find . -maxdepth 3 -name *.py

Defense-in-Depth Layers

L1 Network Isolation — Container communicates only via localhost HTTP
L2 Filesystem Boundary — Overlay FS or scoped mounts, nothing else visible
L3 Command Filtering — Pre-execution validation blocks dangerous patterns
L4 Resource Limits — Iteration caps and timeouts prevent runaway agents

Why This Matters for Enterprises

  • Zero trust by design — AI never gets more access than explicitly granted
  • Audit-friendly — every command is logged with inputs and outputs
  • Ephemeral environments — containers are destroyed post-task, no data leakage
  • Production-safe — read-only bots cannot modify source code under any circumstance

LLM Response Format

{
  "task_done": false,
  "thoughts": "Need to check the file...",
  "command":  "cat /workdir/src/app.py"
}

Loop Termination Conditions

  • LLM sets task_done = true → success
  • Iteration count exceeds max_iterations
  • Elapsed time exceeds timeout_in_seconds
  • Each step: Safety check → Execute → Capture → Feedback

The loop runs autonomously — no human intervention needed. The LLM reasons, acts, observes, and decides when the task is complete.

ReadingBot — Safe Code Analysis

Read-only access with overlay filesystems for zero-risk code inspection

How It Works

  • Extends MicroBot with READ_ONLY permission
  • Uses Docker overlay filesystem — writes go to a temporary upper layer and never touch host files
  • LLM can read, grep, and navigate code freely
  • System prompt instructs: "set task_done=true once the issue is identified"

Automation Use Cases

  • Pre-merge code review in CI pipelines
  • Static bug analysis before hotfix assignment
  • Architecture documentation generation
  • Security vulnerability scanning
from microbots import ReadingBot

bot = ReadingBot(
    model="azure-openai/gpt-5",
    folder_to_mount="/repo/my-project",
)

result = bot.run(
  """Run tests/app_test.py and find
  why the SyntaxError occurs.
  Explain the root cause.""",
  timeout_in_seconds=600,
)

print(result.status)  # True
print(result.result)  # "Line 12: missing
                      #  colon after def..."

WritingBot — Controlled Code Edits

Read-write access with command restrictions for safe file modification

How It Works

  • Extends MicroBot with READ_WRITE permission
  • Direct Docker mount — changes persist to host
  • Restricted to standard Linux tools: sed, awk, grep, git, patch, etc.
  • Blocks non-existent commands like applypatch or edit
  • Instructs LLM to break edits into small, verifiable steps

Automation Use Cases

  • Automated bug fixes in CI/CD pipelines
  • Code refactoring across large codebases
  • Applying patches and resolving merge conflicts
  • Generating boilerplate and test files
from microbots import WritingBot

bot = WritingBot(
    model="azure-openai/gpt-5",
    folder_to_mount="/repo/my-project",
)

result = bot.run(
  """The python script tests/app.py has
  a SyntaxError. Fix the error and
  verify the code runs without errors.
  Then run the test suite.""",
  timeout_in_seconds=600,
)

print(result.status)  # True
print(result.result)
# "Fixed missing colon on line 12.
#  All tests pass."

CustomBot — Build Your Own Agent

Use MicroBot directly with a custom system prompt and mount configuration

from microbots import MicroBot
from microbots.extras.mount import Mount
from microbots.constants import PermissionLabels

# Define custom mount
mount = Mount(
  host_path="/home/user/project",
  sandbox_path="/workdir/project",
  permission=PermissionLabels.READ_WRITE,
)

# Create a fully custom bot
bot = MicroBot(
  model="anthropic/claude-sonnet-4-20250514",
  system_prompt="""You are an expert
  at analyzing Python performance.
  Profile the code and suggest
  optimizations.""",
  folder_to_mount=mount,
)

result = bot.run(
  "Profile and optimize slow functions",
  max_iterations=50,
  timeout_in_seconds=1200,
)

Why CustomBot?

ReadingBot and WritingBot handle common patterns. But when you need a specialized agent—for security auditing, performance profiling, documentation generation, or domain-specific analysis—use MicroBot directly.

What You Control

system_promptFull control over LLM behavior
modelAny supported provider/model
MountCustom path + RO/RW permission
toolsInstall any tool into sandbox
limitsCustom iteration & timeout

Pattern: All built-in bots (ReadingBot, WritingBot, etc.) simply call super().__init__() with pre-configured prompts and mount permissions. You can replicate any of them.

The Tool System

Define, install, and use tools as plugins in any bot's sandbox

@dataclass
class Tool:
  name: str
  description: str

  # Appended to system prompt so
  # the LLM knows how to use it
  usage_instructions_to_llm: str

  # Executed after sandbox starts
  install_commands: List[str]

  # Optional fields
  parameters: Optional[dict]
  env_variables: Optional[List[str]]
  files_to_copy: Optional[List[EnvFileCopies]]
  verify_commands: Optional[List[str]]
  setup_commands: Optional[List[str]]
  uninstall_commands: Optional[List[str]]

Tool Lifecycle

1
Install
Run install_commands, copy env vars & files
2
Verify
Run verify_commands to confirm install
3
Setup
Run setup_commands after code mount
4

LLM uses tool via shell commands

Adding Tools to a Bot

from microbots.tools.tool import (
    parse_tool_definition, Tool
)

# Load from YAML
cscope = parse_tool_definition("cscope.yaml")

# Pass to any bot
bot = WritingBot(
  model="azure-openai/gpt-5",
  folder_to_mount="/repo",
  additional_tools=[cscope],
)

  - apt-get install -y cscope

Web Browsing

  - pip install browser-use playwright
  - playwright install chromium

  - AZURE_OPENAI_API_KEY

Creating Custom Tools

Write a YAML file or define in code — it's that simple

Option A: YAML Definition

# my-linter.yaml
name: eslint
description: JavaScript linter

install_commands:
  - npm install -g eslint

verify_commands:
  - eslint --version

setup_commands:
  - eslint --init

usage_instructions_to_llm: |
  # Run ESLint to find issues:
  #   eslint src/
  # Auto-fix issues:
  #   eslint --fix src/

uninstall_commands:
  - npm uninstall -g eslint

Option B: Programmatic Definition

from microbots.tools.tool import Tool

my_tool = Tool(
  name="pylint",
  description="Python code analyzer",
  install_commands=[
    "pip install pylint",
  ],
  verify_commands=[
    "pylint --version",
  ],
  usage_instructions_to_llm="""
  # Run pylint on a file:
  #   pylint myfile.py
  # Run on entire directory:
  #   pylint src/
  """,
)

bot = WritingBot(
  model="azure-openai/gpt-5",
  folder_to_mount="/repo",
  additional_tools=[my_tool],
)

Pipeline Example: Auto Bug Fix

# automation_pipeline.py
from microbots import ReadingBot, WritingBot

# Step 1: Diagnose the issue
reader = ReadingBot(
  model="azure-openai/gpt-5",
  folder_to_mount="/repo",
)
diagnosis = reader.run(
  "Find why tests/app_test.py fails",
  timeout_in_seconds=300,
)

# Step 2: Fix based on diagnosis
writer = WritingBot(
  model="azure-openai/gpt-5",
  folder_to_mount="/repo",
)
fix = writer.run(
  f"Fix this: {diagnosis.result}",
  timeout_in_seconds=600,
)

Common Automation Patterns

  • PR Review Bot: ReadingBot scans PRs for bugs and security issues on every push
  • Auto-Fixer: WritingBot patches build failures and re-runs CI
  • Log Investigator: LogAnalysisBot analyzes production errors from alerting systems
  • Release Checker: BrowsingBot fetches latest dependency versions for upgrade PRs
  • Custom Pipelines: Chain bots together for multi-stage autonomous workflows

Security Guarantee: Every bot runs in an isolated Docker container with explicit permissions. No shared state, no side effects, no risk to your production environment.
